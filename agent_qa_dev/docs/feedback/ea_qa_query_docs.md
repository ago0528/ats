## 1. 시스템 개요

### 1.1 실행 에이전트(EA)란?

실행 에이전트는 채용 플로우 내에서 사용자의 자연어 요청을 해석하여, 특정 전형의 설정값(RNB)을 변경하거나 액션(안내 발송 등)을 수행하는 AI 에이전트이다.

### 1.2 사용자 페르소나

- **직군**: 중견/대기업 채용 담당자 (인사팀, HR)
- **도구**: ATS(채용솔루션)를 사용하여 공채/수상시 채용을 운영
- **맥락**: 채용 플로우를 세팅하고, 전형별 설정을 관리하며, 지원자 안내 발송 등을 실행
- **특징**: 채용 도메인 전문가이나, ATS의 모든 기능명을 정확히 알지는 못함. 자연어로 의도를 표현

### 1.3 검증용 플로우 세팅

| 순서 | 전형(노드)     | 세부                                    |
| ---- | -------------- | --------------------------------------- |
| 1    | 지원자 접수    | 공고 3개 (예: 디자이너, 개발자, 마케팅) |
| 2    | 역량검사       | 1개                                     |
| 3    | 서류 전형      | 1개                                     |
| 4    | 면접(일정조율) | 1개                                     |
| 5    | 면접(고정)     | 1개                                     |

> 실제 서비스에서는 전형 개수와 순서를 자유롭게 구성 가능하나, QA 검증 시에는 위 구성을 고정

## 2. 플로우 구조 정의

### 2.1 위계 구조

```
채용 플로우
 └─ 전형(노드): 지원자 접수 / 역량검사 / 서류 / 면접(일정조율) / 면접(고정)
     └─ 우측 패널 (노드 클릭 시 열림)
         ├─ 탭 (기본 정보 / 채용분야 / 게시 정보 / 평가 설정 등)
         │   └─ RNB 항목 (설정값)
         └─ 액션 영역
             └─ 액션 항목 (추가 / 수정 / 즉시 실행)
```

### 2.2 전형 순서 규칙

- **고정 순서**: 지원자 접수 → 역량검사 (이 둘의 순서는 반드시 접수 → 역검)
- **자유 배치**: 그 외 전형(서류, 면접 등)은 순서와 개수를 자유롭게 구성 가능
- **전형 간 선행 조건**: 없음 (전형 간 이동에 별도 조건 없음)

### 2.3 RNB와 액션의 차이

| 구분        | RNB                                | 액션                               |
| ----------- | ---------------------------------- | ---------------------------------- |
| **성격**    | 전형의 설정값 (정적 속성)          | 전형에서 실행하는 기능 (동적 동작) |
| **예시**    | 공고명, 접수기간, 합격자 결정 기준 | 합격자 발표 안내 발송, 전형 안내   |
| **조작**    | 값 입력/선택/변경                  | 추가(생성), 수정, 즉시 실행        |
| **UI 위치** | 우측 패널 내 탭 안의 항목          | 우측 패널 내 액션 영역             |

### 2.4 RNB 유형 분류

EA가 처리하는 RNB 항목은 UI 동작 방식에 따라 아래와 같이 분류된다.

| 유형                     | 설명                                | EA 처리 방식                   | 예시                          |
| ------------------------ | ----------------------------------- | ------------------------------ | ----------------------------- |
| **텍스트입력**           | 인풋 필드에 텍스트를 입력           | 값을 생성하여 폼에 채움        | 공고명, 전형명                |
| **날짜입력**             | 날짜 또는 기간을 입력               | 날짜를 생성하여 폼에 채움      | 접수기간, 평가기간            |
| **단일선택**             | 옵션 중 하나를 선택                 | 의도를 파악하여 옵션 변경      | 채용형태, 합격자 결정 기준    |
| **다중선택**             | 옵션을 복수 선택                    | 의도를 파악하여 옵션 다중 변경 | 본인인증                      |
| **버튼클릭**             | 버튼 클릭으로 모달 또는 페이지 이동 | 해당 모달/페이지로 유도        | 평가자 세부 설정, 지원서 설정 |
| **옵션 변경 + 버튼클릭** | 토글 ON 후 모달 띄움                | ON 변경 후 모달 유도           | 지원서 블라인드, 면접비 지급  |
| **예외케이스**           | 별도 로직 필요                      | 전용 로직 처리                 | 채용 분야 설정                |

### 2.5 액션 동작 분류

| 동작           | 설명                    | 공채 | 수상시                    |
| -------------- | ----------------------- | ---- | ------------------------- |
| **추가(생성)** | 새 액션을 생성          | ✅   | ❌ (정책 미비, 추후 추가) |
| **수정**       | 기존 액션의 내용을 수정 | ✅   | ✅                        |
| **즉시 실행**  | 액션을 바로 실행        | ✅   | ✅                        |

- **공채**: 스케줄(예약) 실행 가능
- **수상시**: 트리거 기반 실행이 주 (동시다발 전형 진행 특성)
- **공통**: 즉시 실행 지원

## 6. 질의 분류 체계

### 6.1 분류 구조

```
Level 1: 대분류
  ├─ RNB (설정값 변경)
  └─ 액션 (기능 실행)

Level 2: 타깃 항목
  ├─ RNB: 구체적 항목명 (예: 공고명, 접수기간)
  └─ 액션: 구체적 액션명 (예: 합격자 발표)

Level 3: 질의 변형
  ├─ 타깃 특정 (전형/공고를 명시) → ACTION 응답 기대
  └─ 타깃 비특정 (전형/공고를 생략) → SELECT 응답 기대

Level 4 (액션 전용): 동작 분류
  ├─ 추가(생성) 요청
  ├─ 수정 요청
  └─ 즉시 실행 요청
```

### 6.2 질의 ID 규칙

```
RNB-{전형코드}-{항목번호}-{변형코드}

예시:
RNB-APP-01-S  → 지원서접수 > 공고명 > 타깃 특정
RNB-APP-01-M  → 지원서접수 > 공고명 > 타깃 비특정(멀티셀렉트)

ACT-{액션코드}-{전형코드}-{동작코드}-{변형코드}

예시:
ACT-A04-DOC-C-S  → 합격자발표 > 서류전형 > 추가(Create) > 타깃 특정
ACT-A04-ALL-E-M  → 합격자발표 > 전형 미지정 > 즉시실행(Execute) > 타깃 비특정
```

**전형 코드**: APP(지원서접수), ACC(역량검사), DOC(서류), INT(면접), ITC(면접-일정조율), ALL(미지정)
**동작 코드**: C(추가/Create), U(수정/Update), E(즉시실행/Execute)
**변형 코드**: S(타깃 특정/Specific), M(타깃 비특정/Multi-select)

## 8. 질의 생성 프롬프트

> 아래 프롬프트를 LLM에 제공하여 질의 풀을 체계적으로 생성할 수 있다.

### 8.1 시스템 프롬프트

```
당신은 중견/대기업 채용 담당자(HR)의 발화를 시뮬레이션하는 전문가입니다.

## 배경
- 사용자는 ATS(채용솔루션)를 사용하여 공채 채용을 운영하는 채용 담당자입니다.
- 이 ATS에는 AI 에이전트(실행 에이전트)가 탑재되어 있어, 자연어로 채용 플로우의 설정을 변경하거나 안내 발송 등의 액션을 실행할 수 있습니다.
- 사용자는 채용 도메인 전문가이지만, ATS의 기능명이나 데이터 키를 정확히 알지는 못합니다. 일상적이고 자연스러운 표현을 사용합니다.

## 채용 플로우 구조
- 전형 순서: 지원자 접수(공고 3개: 디자이너/개발자/마케팅) → 역량검사 → 서류 → 면접(일정조율) → 면접(고정)
- 각 전형은 우측 패널에 RNB(설정값)와 액션(실행 기능)을 가집니다.

## 질의 생성 규칙

### RNB 질의
- 각 RNB 항목에 대해 2가지 변형을 생성합니다:
  1. **타깃 특정**: 전형/공고를 명시하여 즉시 적용 가능한 질의
  2. **타깃 비특정**: 전형/공고를 생략하여 선택이 필요한 질의

### 액션 질의
- 각 액션에 대해 동작(추가/수정/실행) × 변형(특정/비특정) 조합을 생성합니다.
- 여러 전형에서 공유되는 액션은, 전형 특정 질의와 전형 미지정 질의를 모두 포함합니다.

### 표현 가이드라인
- 실제 채용 담당자가 업무 중 입력할 법한 자연스러운 문장을 사용합니다.
- 격식체/비격식체를 혼용합니다. (예: "~해줘", "~할게요", "~변경 부탁", "~좀 바꿔줘")
- ATS 기능의 정확한 명칭 대신, 의미 기반의 자연어를 사용합니다.
  - 좋은 예: "서류 평가 기간 좀 늘려줘", "합격자한테 메일 보내야 하는데"
  - 나쁜 예: "SCREENING_EVALUATION_PERIOD를 변경해줘", "합격자 발표 액션을 추가해줘"
- 다만, 일부 질의는 정확한 기능명을 아는 사용자의 입력도 포함합니다.
  - 예: "지원서 블라인드 설정 켜줘", "오픈그래프 제목 수정해줘"
- 맥락 정보를 포함하는 질의도 생성합니다.
  - 예: "내일 접수 마감인데, 마감일을 3일 연장해줘"
  - 예: "면접관들한테 다음 주 면접 일정 안내 보내야 하는데"

```
